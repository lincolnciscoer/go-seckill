# 第 02 次提交：接入配置系统、日志、优雅退出和 Swagger 骨架

## 本次目标

把项目从“能跑的最小服务”升级成“具备基础工程能力的服务”。

这一步主要解决这些问题：

- 启动参数不能一直写死在代码里
- 日志不能只靠零散的 `fmt.Println`
- 关闭服务时不能直接粗暴退出
- API 返回格式需要开始统一
- 接口文档需要有基础展示入口

## 我这次做了什么

1. 增加了配置系统
   - 支持从 `yaml` 文件读取配置
   - 支持环境变量覆盖配置
   - 提供默认值，避免最小开发场景下配置缺失
2. 增加了结构化日志
   - 接入 `zap`
   - 请求访问日志改成结构化输出
3. 增加了 panic 恢复中间件
   - 避免单个请求异常把服务带崩
4. 实现了优雅退出
   - 响应 `SIGINT` / `SIGTERM`
   - 给正在处理的请求留出收尾时间
5. 统一了 HTTP 响应结构
   - 增加 `code`、`message`、`data` 外层格式
   - 建立错误码骨架
6. 接入了 Swagger 骨架
   - 提供 `/swagger/index.html`
   - 为健康检查接口生成基础文档

## 为什么这一步很重要

这一提交虽然还没有进入数据库、Redis 或秒杀核心逻辑，但它决定了项目后面会不会“越写越乱”。

### 配置系统的意义

如果端口、环境、日志级别全部写死在代码里，后面接入 MySQL、Redis、RocketMQ 时，启动参数会迅速失控。

### 结构化日志的意义

日志不是只给“现在能看懂的人”看的，而是为了后面排查问题、分析请求、定位慢点。

结构化日志最大的价值是：字段可以被机器理解，而不是只能靠肉眼扫文本。

### 优雅退出的意义

Web 服务关闭时，最怕的情况是：

- 新请求还在进来
- 老请求做了一半
- 服务直接断掉

优雅退出就是为了让服务在收到停止信号后，尽可能有序地停止工作。

### 统一响应的意义

如果每个接口都返回不同格式，后面前后端联调、压测、错误分析都会越来越痛苦。

所以从第 2 次提交开始，我们就固定一个统一的外层响应格式。

## 关键代码入口

  - 负责结构化日志实例创建
- `internal/transport/http/middleware/`

- `internal/transport/http/response/response.go`

- `cmd/api/main.go`
  - 负责服务启动、依赖组装、优雅退出
- `internal/transport/http/router/router.go`
  - 负责注册业务路由和 Swagger 路由

## 你现在应该掌握什么

### 1. 配置优先级

我们当前采用的是：

`配置文件 -> 环境变量覆盖`

这是一种非常常见的工程实践。

### 2. 为什么要在路由层做中间件

请求日志、恢复、鉴权、限流这些能力，本质上都不是某个单独业务接口的责任。

它们更适合放在“进入业务之前”的中间层处理。

### 3. 为什么要尽早统一响应格式

因为接口一旦多起来，再回头统一返回结构的成本会明显变高。
