# go-seckill 简历项目与面试话术版

## 1. 简历项目版

### 项目名称

Go 高并发电商秒杀系统后端

### 一句话项目描述

基于 `Gin + MySQL + Redis + Lua + RocketMQ` 设计并实现高并发电商秒杀系统，完成秒杀准入、库存预扣减、异步下单、消费幂等、限流保护与可观测体系建设。

### 简历版项目描述

负责设计并实现一个高并发电商秒杀系统后端，采用 `Gin` 构建 HTTP 服务，使用 `MySQL` 作为最终一致性存储，基于 `Redis + Lua` 实现秒杀准入与库存原子预扣减，结合 `RocketMQ` 实现异步下单与消费幂等，补充 `JWT` 鉴权、活动预热、限流防重复提交、Prometheus/Grafana/Jaeger/pprof 可观测与压测能力，形成可运行、可展示、可压测的完整后端项目。

### 简历亮点版写法

- 使用 `Redis + Lua` 将秒杀入口前置到缓存层，原子完成活动状态校验、时间窗口校验、一人一单判断与库存预扣减，降低数据库在高并发入口处的压力。
- 引入 `RocketMQ` 将同步下单链路改造为“准入入队 + consumer 异步落库”，并通过 `mq_consume_logs` 表和消息 ID 唯一约束实现消费幂等，避免重复消费造成重复扣库存。
- 秒杀核心写路径采用原生 SQL，在事务中完成库存扣减与订单创建；普通管理链路采用 `GORM`，在开发效率和热点路径可控性之间做分层取舍。
- 实现用户 / IP / 活动维度限流与短时重复提交保护，异常场景下返回明确业务码，如 `RATE_LIMITED`、`ORDER_PROCESSING`、`SOLD_OUT`。
- 接入 `Prometheus / Grafana / OpenTelemetry / Jaeger / pprof`，完成 HTTP 指标、秒杀结果指标、MQ 消费指标、订单状态流转指标、链路追踪与性能分析入口建设。
- 编写仓库内置压测工具 `cmd/loadtest`，支持自动注册用户、创建活动、预热缓存、并发发起秒杀请求并输出延迟和结果统计。
- 针对热点活动详情链路，补充空值缓存、TTL 随机抖动与 `singleflight` 缓存重建保护，分别解决缓存穿透、缓存雪崩和缓存击穿问题。

### 适合简历的 3 条精炼 bullet

- 基于 `Redis + Lua` 实现秒杀准入与库存原子预扣减，支持活动状态校验、时间窗口校验和一人一单判断，降低数据库入口压力。
- 基于 `RocketMQ` 实现异步下单，使用消息 ID 幂等表防止重复消费，完成“秒杀请求入队 -> consumer 落库 -> 订单查询”闭环。
- 接入 `Prometheus / Grafana / Jaeger / pprof` 与内置压测工具，支持 HTTP 指标、MQ 消费指标、链路追踪和性能分析。
- 通过空值缓存、TTL 随机抖动和 `singleflight` 热点重建保护，补齐三类典型缓存问题治理。

## 2. 面试话术版

### 30 秒版本

我做的是一个 Go 电商秒杀系统，核心链路是 `Gin API -> Redis Lua 准入 -> RocketMQ 入队 -> consumer 异步落库 -> MySQL 最终一致`。其中 `Redis + Lua` 负责高并发下的原子校验和库存预扣减，`RocketMQ` 负责削峰填谷和异步下单，数据库热点写路径我用原生 SQL 控制事务和锁，可观测上接了 Prometheus、Jaeger 和 pprof。

### 1 分钟版本

这个项目一开始是同步版秒杀，我先把商品、活动、订单、登录这些基本能力搭起来，再逐步演进成高并发版本。后面我把秒杀入口前置到 `Redis + Lua`，在缓存层原子完成活动有效性判断、库存判断和一人一单判断，只有通过准入的请求才会继续往后走。再往后我把下单改造成 `RocketMQ` 异步消费，API 只负责返回 `queued`，consumer 再异步落库，数据库通过原生 SQL 在事务里扣库存和创建订单，并用消息 ID 幂等表避免重复消费。最后又补了限流、防重复提交、订单处理中状态、Prometheus/Grafana/Jaeger/pprof 和压测工具，让它更像一个完整后端项目。

### 3 分钟版本

我这个秒杀系统主要解决的是高并发下的几个核心问题：防超卖、一人一单、削峰填谷、消费幂等和可观测性。

整体架构上，对外是 `Gin` 提供 HTTP API，普通管理链路比如商品和活动管理用 `GORM` 提升开发效率；真正的热点写链路，比如扣库存和建订单，用原生 SQL 保证事务边界和锁行为可控。数据层上 `MySQL` 负责最终落库，`Redis` 负责活动预热、库存缓存、订单处理中状态和限流计数。

秒杀入口上，我没有让请求直接打数据库，而是先走 `Redis + Lua`。Lua 脚本会原子地做几件事：判断活动状态是否可用、活动是否开始或结束、用户是否已经下过单、库存是否足够，以及在通过准入时先做库存预扣减并打用户标记。这样可以把大量无效请求挡在数据库前面，同时避免并发条件下的超卖问题。

通过准入之后，我又把同步下单改造成 `RocketMQ` 异步下单。API 在这一层只返回 `queued` 和 `order_no`，consumer 独立消费消息后才真正落库。这样做的好处是把瞬时流量削成了消息队列中的平滑消费流。consumer 里我先写一条消息幂等日志，再执行库存扣减和订单创建事务，避免重复消息造成二次扣库存。订单查询这边，如果用户马上查订单而 consumer 还没落库，我会先从 Redis 的订单状态缓存里返回 `ORDER_PROCESSING`，而不是直接 404。

工程上我还补了 `Prometheus / Grafana / Jaeger / pprof`，把 HTTP 请求、秒杀结果、MQ 消费结果、订单状态流转都做成指标，同时把 API 和 consumer 都接入 trace，这样压测后可以直接看指标、看链路、看性能热点。最后我还写了仓库内置压测工具，可以直接一键创建用户、活动并发起压测，输出 QPS 结果和延迟统计。

## 3. 高频追问回答

### Q1：为什么秒杀入口要用 Redis + Lua？

因为秒杀入口最大的特点是瞬时流量很高，如果所有请求直接打 MySQL，数据库会很快成为瓶颈。`Redis` 性能更高，`Lua` 又能保证脚本原子执行，所以适合把活动状态校验、库存判断和一人一单判断前置到缓存层完成。

### Q2：Lua 脚本里具体做了什么？

主要做五件事：

1. 校验活动状态是否可用
2. 校验当前时间是否在活动窗口内
3. 判断用户是否已经下过单
4. 判断库存是否大于 0
5. 对通过准入的请求执行库存预扣减并写用户标记

### Q3：怎么保证不超卖？

第一层是在 Redis 里通过 Lua 原子预扣减库存；第二层是在数据库里通过事务更新库存和写订单。即使极端情况下 Redis 和数据库状态短时间不一致，也会在数据库侧兜底，失败时再对 Redis 做补偿。

### Q4：为什么还要引入 RocketMQ？

因为 Redis + Lua 只是解决了“谁能进来”的问题，真正的下单写库如果仍然同步执行，在高并发下数据库压力还是会很大。`RocketMQ` 可以把通过准入的请求先排队，API 快速返回，consumer 按可控速度落库，从而实现削峰填谷。

### Q5：消息重复消费怎么处理？

我在数据库里加了 `mq_consume_logs` 表，用 `message_id` 做唯一约束。consumer 每次消费时先在事务最前面插入这条记录，如果插入失败说明这条消息已经处理过，就直接返回。这样可以保证重复消费不会重复扣库存。

### Q6：为什么普通链路用 GORM，热点链路用原生 SQL？

因为普通链路更看重开发效率和代码维护成本，比如商品、活动管理这类 CRUD，用 GORM 很合适；热点写链路更看重事务边界、SQL 执行计划和锁行为可控，所以我在订单创建和库存扣减这类路径上用了原生 SQL。

### Q7：限流做了哪些维度？

做了三层：

1. 用户维度限流
2. IP 维度限流
3. 活动维度限流

另外还加了一个短时重复提交保护锁，避免用户在极短时间内连续点击造成重复入队。

### Q8：为什么订单详情要返回 `ORDER_PROCESSING`？

因为异步下单场景里，API 返回成功后 consumer 还没落库是正常情况。如果这时候直接返回 404，用户会误以为下单失败。所以我用 Redis 缓存了一份订单状态，在订单真正落库前先返回 `202 ORDER_PROCESSING`。

### Q9：你这个项目怎么体现工程能力？

我觉得主要体现在四点：

1. 代码是按阶段演进的，不是一口气堆出来
2. 普通链路和热点链路做了分层取舍
3. 异常场景和幂等问题都考虑进去了
4. 有完整的可观测和压测能力，不是只写功能

### Q10：你做了哪些缓存保护手段？

我不是只把活动数据放进 Redis 就结束了，还专门补了三类典型缓存问题治理：

1. 用空值缓存解决缓存穿透
2. 用 TTL 随机抖动解决缓存雪崩
3. 用 `singleflight` 合并热点 key 的缓存重建请求，解决缓存击穿

这样热点活动详情在高并发场景下，不会因为不存在 ID、大量 key 同时过期或者单个热点 key 失效而直接打穿数据库。

## 4. 项目亮点总结

如果面试官让我总结亮点，我会这么说：

这个项目的亮点不只是“用了很多中间件”，而是我把秒杀系统最核心的几个问题串成了一条完整演进链路：先做同步版闭环，再用 `Redis + Lua` 做高并发准入，再用 `RocketMQ` 做异步下单和消费幂等，最后补上限流、状态缓存、可观测和压测工具，让它从 Demo 逐步长成一个更像真实后端系统的项目。
