# 第 01 次提交：初始化仓库并跑通最小服务

## 本次目标

从空目录开始，完成一个最小但规范的 Go Web 项目骨架，并让它具备：

- Git 仓库管理能力
- Go 模块管理能力
- 最小 HTTP 服务能力
- 健康检查接口
- 基础测试
- 后续可扩展的目录结构

## 我这次做了什么

1. 初始化了 Git 仓库，并把默认分支设置为 `main`
2. 初始化了 Go 模块 `go-seckill`
3. 创建了基础目录结构：
   - `cmd/api`：程序启动入口
   - `internal/transport/http`：HTTP 路由和处理器
   - `configs`：配置样例
   - `docs/commit-notes`：每次提交对应的教学文档
   - `pkg`：未来放可复用公共代码
4. 接入了最小 Gin 服务
5. 提供了 `GET /healthz` 健康检查接口
6. 为健康检查接口补上了自动化测试
7. 编写了根 `README.md`，说明项目定位和当前启动方式

## 为什么第一步只做这些

因为你是第一次完整做后端项目，第一步最重要的不是“功能多”，而是“脚手架稳”。

如果一开始就把 MySQL、Redis、JWT、RocketMQ 一起塞进来，会同时出现下面几类问题：

- 目录结构混乱，不知道代码应该放哪
- 启动流程不清楚，不知道程序是怎么跑起来的
- 出错时定位困难，不知道是框架问题、配置问题还是业务问题
- 学习负担太重，容易从第一步就失去掌控感

所以我们第一步只做一件事：把最小服务跑起来，并且保证它已经是一个“以后可以持续长大”的项目，而不是随手写的单文件 demo。

## 关键代码入口

- `cmd/api/main.go`
  - 程序主入口
  - 负责启动 Gin 服务
  - 暂时通过环境变量 `APP_PORT` 控制端口
- `internal/transport/http/router/router.go`
  - 路由注册入口
  - 后续新增接口时，会继续在这里组织模块路由
- `internal/transport/http/handler/health.go`
  - 健康检查接口处理逻辑
- `internal/transport/http/handler/health_test.go`
  - 健康检查接口的基础测试

## 你现在应该掌握什么

完成这一提交后，你应该理解这些基础概念：

### 1. `cmd` 和 `internal` 为什么要分开

- `cmd` 放程序入口
- `internal` 放项目内部实现

这样做的好处是职责清晰，后面如果我们新增 `cmd/consumer` 做 RocketMQ 消费者，也不会乱。

### 2. 为什么一开始就写测试

很多初学者会把测试放到最后，但项目一旦复杂起来，最后补测试会非常痛苦。

从第一步开始就建立“写功能就验证”的习惯，后面你实现登录、订单、秒杀时会非常受用。

### 3. `GET /healthz` 的价值是什么

健康检查接口不是为了“炫技”，而是为了回答一个非常基础的问题：

> 这个服务现在有没有正常活着？

以后部署、压测、接监控时，这个接口都会被频繁用到。

## 本次如何测试

### 1. 自动化测试

在项目根目录执行：

```bash
go test ./...
```

预期：

- 测试通过
- 说明健康检查接口的最小行为符合预期

### 2. 手动启动服务

在项目根目录执行：

```bash
go run ./cmd/api
```

### 3. 手动验证接口

打开新的终端执行：

```bash
curl http://127.0.0.1:8080/healthz
```

预期返回类似：

```json
{
  "status": "ok",
  "service": "go-seckill",
  "time": "2026-04-17T00:00:00Z"
}
```
