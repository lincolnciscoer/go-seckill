## 1. 项目总览

### Q1：你这个项目是做什么的？

这是一个 Go 电商秒杀系统后端项目，核心目标是解决高并发场景下的库存扣减、一人一单、削峰填谷、异步下单、消费幂等和可观测性问题。整体链路是 `Gin API -> Redis Lua 准入 -> RocketMQ 入队 -> consumer 异步落库 -> MySQL 最终一致`。

### Q2：这个项目为什么适合作为秋招后端项目？

因为它不只是普通 CRUD，而是覆盖了后端岗位常见的核心能力：接口设计、数据库建模、缓存、并发控制、消息队列、幂等、限流、观测和压测。并且技术选型都比较主流，容易和真实业务结合。

### Q3：你在这个项目里主要负责什么？

我负责整个项目的设计和实现，包括系统分层、数据库表设计、Redis + Lua 秒杀准入、RocketMQ 异步下单、订单幂等、限流策略、缓存治理、Prometheus / Jaeger / pprof 可观测，以及压测工具和项目文档。

### Q4：项目现在做到了什么程度？

目前已经完成登录鉴权、商品管理、活动管理、活动预热、同步版秒杀闭环、Redis Lua 秒杀准入、RocketMQ 异步下单、消费幂等、限流、订单处理中状态缓存、Prometheus/Grafana/Jaeger/pprof 和仓库内置压测工具。

### Q5：这个项目的核心亮点是什么？

我觉得有四个核心亮点：

1. 秒杀入口前置到 `Redis + Lua`，解决高并发下的原子校验和库存预扣减。
2. 下单链路异步化，利用 `RocketMQ` 做削峰填谷。
3. 消费端通过消息 ID 幂等表避免重复消费造成重复扣库存。
4. 补齐了限流、缓存治理、可观测和压测，让项目更像真实系统。

## 2. 架构设计

### Q6：整体架构是怎样的？

对外是 Gin 提供 HTTP API，普通管理链路走 GORM，热点写链路走原生 SQL。活动和库存等热点数据放 Redis，秒杀准入放到 Lua 脚本中原子执行。通过准入的请求写入 RocketMQ，由 consumer 异步消费并最终落到 MySQL。观测上接了 Prometheus、Grafana、Jaeger 和 pprof。

### Q7：为什么没有一开始就拆微服务？

因为校招项目更重要的是把一条主链路做深，而不是把系统拆碎。当前我采用的是“模块化单体 + 独立 consumer 进程”的方式，既保留了工程复杂度，又不会引入过多服务治理成本。

### Q8：为什么 API 和 consumer 要拆成两个入口？

因为它们职责完全不同。API 面向请求接入和快速响应，consumer 面向消息消费和异步落库。拆成 `cmd/api` 和 `cmd/consumer` 之后，部署、调试、日志和指标都会更清晰。

### Q9：为什么 MySQL 还是要保留？

Redis 和 MQ 只能帮我提高并发能力和解耦，但最终一致性的数据还是要落到关系型数据库里。订单、活动、库存最终状态必须有可靠存储，MySQL 适合做最终一致性和事务兜底层。

### Q10：为什么 Redis 里既存活动详情，又存库存？

因为两者用途不同。活动详情用于热点查询和预热，库存用于秒杀准入和高并发判断。库存 key 通常比详情 key 更敏感，因为它直接参与秒杀脚本和实时库存视图更新。

## 3. 技术选型

### Q11：为什么 Web 框架选 Gin？

Gin 轻量、性能好、生态成熟，做 API 服务很合适。对于这个项目来说，我需要的主要是路由、中间件、JSON 绑定和恢复机制，Gin 足够，而且面试官也普遍熟悉。

### Q12：为什么普通链路用 GORM，热点链路用原生 SQL？

因为普通链路更重视开发效率，比如商品、活动管理这种 CRUD，用 GORM 很方便。热点链路比如库存扣减和订单创建，更重视事务边界、SQL 控制能力和锁行为可控，所以我用原生 SQL。

### Q13：为什么消息队列选 RocketMQ？

RocketMQ 在订单、交易、电商场景里非常常见，适合做削峰填谷、异步解耦和最终一致性。这个项目里它主要负责把“通过秒杀准入的请求”转成异步下单任务。

### Q14：为什么 JWT 放在这个项目里？

因为秒杀系统一定要知道“当前用户是谁”，不然后面的一人一单、订单归属、订单查询都没法做。JWT 在这里主要是解决鉴权和身份识别。

### Q15：为什么还接 Prometheus / Grafana / Jaeger / pprof？

因为一个高并发项目如果没有指标、链路和性能分析能力，面试时说服力会差很多。接了这些之后，我可以真正展示系统的 QPS、延迟、秒杀结果、MQ 消费结果，以及 API/consumer 的 trace 和性能热点。

## 4. 秒杀主链路

### Q16：用户发起秒杀请求后，系统做了什么？

先经过 JWT 鉴权，再经过限流和重复提交保护，然后进入 Redis Lua 脚本做原子校验。脚本判断活动状态、时间窗口、重复下单和库存是否足够，通过后先在 Redis 里预扣减库存并写用户标记。API 随后把下单消息发到 RocketMQ，并返回 `queued`。consumer 再异步落库。

### Q17：为什么秒杀接口现在返回的是 `queued` 而不是直接返回订单成功？

因为现在下单链路已经异步化了。API 只负责“准入 + 入队”，真正的订单创建是 consumer 异步完成的，所以返回 `queued` 更符合真实状态。

### Q18：为什么要设计订单处理中状态？

异步系统里，用户收到 `queued` 后立刻查订单是很正常的。如果这时 consumer 还没落库，直接返回 404 会让用户误以为失败。所以我在 Redis 里缓存订单状态，返回 `202 ORDER_PROCESSING`。

### Q19：同步版秒杀和异步版秒杀的区别是什么？

同步版里，通过准入后直接走数据库事务扣库存和创建订单，链路简单但数据库压力更大。异步版里，通过准入后只负责入队，consumer 再异步落库，数据库压力更平滑。

### Q20：为什么项目里还保留同步版秒杀思路？

因为它是系统演进过程的一部分，也是面试时解释“为什么要引入 MQ”的基础。没有同步版，你很难说明异步化到底优化了什么。

## 5. Redis + Lua

### Q21：为什么秒杀入口要用 Redis + Lua？

因为秒杀入口最大的特点是瞬时并发高，数据库不适合承接所有请求。Redis 单线程模型加 Lua 原子执行，适合做活动校验、库存判断和一人一单这种强并发前置判断。

### Q22：Lua 脚本里做了哪些事情？

主要做这些：

1. 校验活动状态是否可用
2. 校验当前时间是否在活动窗口内
3. 判断用户是否已经秒杀过
4. 判断库存是否大于 0
5. 扣减库存并写用户标记

### Q23：Lua 脚本为什么能防超卖？

因为它在 Redis 内部原子执行，库存判断和扣减之间不会被其他请求打断。这样可以避免多个请求同时读到相同库存后都去扣减，导致超卖。

### Q24：如果消息发送失败怎么办？

我在 API 侧会对 Redis 做补偿，也就是把预扣减的库存加回去，并删除用户标记。这样可以避免“库存被扣了，但消息没发出去”造成的不一致。

### Q25：为什么库存 key 和活动详情 key 要分开？

因为库存是秒杀实时状态，变化频率高；活动详情相对稳定，变化频率低。分开之后可以分别失效、分别覆盖，结构更清晰。

## 6. RocketMQ 与异步下单

### Q26：为什么要引入 RocketMQ？

引入 RocketMQ 的核心目的是削峰填谷。通过 Redis Lua 准入的请求虽然数量已经少很多，但直接同步写 MySQL 还是会让数据库承压。RocketMQ 把这些请求变成异步任务，consumer 可以按可控速度落库。

### Q27：RocketMQ 在项目里承担什么职责？

主要承担三件事：

1. API 与数据库写链路解耦
2. 将突发请求削峰为平滑消费流
3. 为后续消费重试、幂等和最终一致性提供基础

### Q28：为什么用 simple consumer？

因为当前项目的目标是先把异步订单消费链路跑通，simple consumer 模式更容易理解，便于直观看到 `Receive -> Ack` 的消费流程。

### Q29：消息体里带了哪些信息？

主要有：

- `order_no`
- `user_id`
- `activity_id`
- `product_id`
- `quantity`
- `amount`
- `status`
- `created_at`

这些字段足够支撑 consumer 异步落库。

### Q30：为什么消息里还要带 orderNo？

因为 `orderNo` 是订单业务主键之一，API 在入队时就先生成它，可以让用户在异步阶段立即拿到订单编号，后续查询也依赖这个编号。

## 7. 幂等与一致性

### Q31：消息重复消费怎么处理？

我在 MySQL 里设计了 `mq_consume_logs` 表，用 `message_id` 做唯一约束。consumer 每次处理消息时先写这张表，如果写失败说明消息已经处理过，直接返回。

### Q32：为什么消费幂等要放在事务最前面？

如果先扣库存再判断消息是否处理过，重复消息会导致库存重复扣减。这是很危险的。所以正确做法是先做消息幂等，再进入扣库存和建订单事务。

### Q33：数据库里还有哪些兜底约束？

订单表里我还保留了 `user_id + activity_id` 唯一约束，它能从业务层面兜底一人一单。这样即使上层判断漏掉，数据库仍然能防止重复订单。

### Q34：Redis 和 MySQL 不一致怎么办？

当前策略是“Redis 前置准入，MySQL 最终一致”。如果在 API 入队前或消息发送失败，我会做 Redis 补偿；如果 consumer 落库失败，则消息会重试，最终以 MySQL 成功提交为准。

### Q35：为什么说这个项目是最终一致，而不是强一致？

因为 Redis 准入、MQ 入队和 MySQL 落库是分阶段完成的，中间会有短暂不一致窗口，比如订单处于 `queued` 但还没落库。这种场景下的目标是最终状态一致，而不是每个瞬间都完全一致。

## 8. 限流与防重复提交

### Q36：你做了哪些限流？

做了三层：

1. 用户维度限流
2. IP 维度限流
3. 活动维度限流

### Q37：为什么限流要分多个维度？

因为单一维度不够。只按用户限流挡不住恶意 IP，只按 IP 限流挡不住同一个用户重复打，只按活动限流又可能误伤所有人。多维度组合能更稳妥。

### Q38：重复提交保护和一人一单有什么区别？

重复提交保护是短时间内的接口层防抖，防的是用户瞬间连点、多次入队；一人一单是业务约束，防的是整个活动周期内一个用户只能下一个订单。

### Q39：为什么限流返回 429 很重要？

因为这说明系统在超压时是主动保护自己的，而不是等数据库被打爆再崩掉。从项目展示角度，能说明你考虑了高并发保护策略。

## 9. 缓存策略与三类缓存问题

### Q40：你项目里的热点缓存策略是什么？

活动列表和活动详情采用缓存优先、数据库兜底的方式，支持活动预热；库存单独存 key，便于 Lua 脚本和实时库存视图使用；订单状态也缓存到 Redis，用于异步链路下的状态查询。

### Q41：你解决了哪些缓存问题？

当前已经补了三类典型问题：

1. 缓存穿透：空值缓存
2. 缓存雪崩：TTL 随机抖动
3. 缓存击穿：`singleflight` 合并热点 key 重建请求

### Q42：缓存穿透是什么？你怎么解决？

缓存穿透是指请求查一个根本不存在的数据，缓存一直 miss，最终每次都打数据库。我在活动详情缓存里加了空值缓存，对不存在的活动 ID 写一个短 TTL 的空标记。

### Q43：缓存雪崩是什么？你怎么解决？

缓存雪崩是指很多 key 使用相同 TTL，在同一时间一起失效，导致大量请求同时回源。我在活动详情、活动列表、库存 key 上增加了 TTL 随机抖动，让过期时间错开。

### Q44：缓存击穿是什么？你怎么解决？

缓存击穿是指某个热点 key 刚好过期，很多请求同时 miss 并打到数据库。我在活动列表和详情的重建路径中用了 `singleflight`，同一个 key 的并发请求只会触发一次数据库查询。

### Q45：为什么现在没上 Bloom Filter？

因为当前活动 ID 的查询规模和业务复杂度下，空值缓存已经能解决主要的穿透问题。Bloom Filter 可以作为下一步增强项，但不是当前最优先。

### Q46：为什么还做了活动预热？

预热可以把活动详情、活动列表、库存 key 提前打进 Redis，避免活动刚开始时所有请求第一次都去打数据库。

## 10. 数据库与 SQL

### Q47：订单创建时数据库事务做了哪些操作？

在事务里主要做：

1. 消费幂等记录写入
2. 更新活动库存
3. 插入订单
4. 提交事务

### Q48：库存扣减 SQL 为什么用 `available_stock >= 1` 条件？

因为这是数据库层最后一道防线。如果并发极端情况下 Redis 判断失效或消息重试导致竞争，数据库这条 SQL 也能防止把库存扣成负数。

### Q49：为什么订单表要设计 `order_no`？

因为订单号是对外暴露的稳定标识，用户查询、日志排查、消息关联都更适合用订单号，而不是直接用自增 ID。

### Q50：你这个项目里有哪些关键表？

主要有：

- `users`
- `products`
- `seckill_activities`
- `seckill_stocks`
- `orders`
- `mq_consume_logs`

## 11. 可观测性

### Q51：你项目里有哪些监控指标？

目前有三类：

1. HTTP 请求指标
2. 秒杀结果指标
3. MQ 消费结果指标
4. 订单状态流转指标

### Q52：为什么要接 Jaeger？

因为 Jaeger 能把 API 和 consumer 的链路串起来。比如一次秒杀请求从 API 入队，到 consumer 落库，这条链可以在 trace 里看到服务边界。

### Q53：pprof 在这个项目里有什么价值？

pprof 主要是给性能分析和压测复盘用，比如查看 goroutine、heap、mutex、CPU profile。它能证明你不是只会写接口，也会定位性能瓶颈。

### Q54：Grafana 面板里你最关注什么？

我最关注：

- HTTP QPS
- HTTP P95 延迟
- 秒杀结果分布
- MQ 消费结果分布

这些能比较直观地反映系统运行状态。

## 12. 压测

### Q55：你怎么做压测的？

我没有依赖外部压测工具，而是写了仓库内置的 `cmd/loadtest`，它能自动注册用户、创建商品和活动、预热缓存，再并发发起秒杀请求并输出统计结果。

### Q56：你压出了什么结果？

我做了两类样例：

1. 正常区间：
   - `users=20`
   - `concurrency=10`
   - `stock=20`
   - `20/20` 返回 `OK`
2. 触发限流：
   - `users=40`
   - `concurrency=20`
   - `stock=20`
   - `20` 个 `OK`
   - `20` 个 `RATE_LIMITED`

### Q57：如果面试官说并发量不够大怎么办？

我会明确说，这两组数据是当前本地单机开发环境下的阶段性验证结果，主要用于验证异步链路和限流保护是否生效，不夸大成线上吞吐能力。重点在于结果能说明系统具备“正常工作”和“超压保护”两种行为。

### Q58：你会怎么进一步提升压测说服力？

可以继续补三类对比：

1. 同步版 vs Lua+MQ 异步版
2. 有限流 vs 无限制流
3. 有缓存治理 vs 无缓存治理

## 13. 部署与运行

### Q59：为什么用 Docker Compose？

因为这个项目依赖 MySQL、Redis、RocketMQ、Prometheus、Grafana、Jaeger，本地一键拉起比手工安装更稳定，也更方便展示项目完整度。

### Q60：Windows + Docker Desktop 里踩过什么坑？

踩过两个典型问题：

1. `host.docker.internal` 和 `127.0.0.1` 的连接差异
2. RocketMQ 镜像和本地 Docker 网络问题

这些我最后都在配置和本地文档里写清楚了。

## 14. 深挖追问

### Q61：如果 RocketMQ 挂了怎么办？

当前 API 在消息发送失败时会回滚 Redis 里的库存和用户标记，避免用户“准入成功但消息没出去”。如果要进一步增强，可以引入本地消息表或事务消息方案。

### Q62：为什么订单状态缓存不是直接放数据库里查？

因为异步链路里，订单还没落库时数据库查不到，但用户又需要明确反馈。所以我用 Redis 缓存一个轻量状态，在“排队中”阶段更适合。

### Q63：如果 Redis 挂了怎么办？

当前秒杀入口强依赖 Redis，所以 Redis 故障时秒杀能力会受影响。这是典型的高性能换高依赖设计。更进一步可以考虑 Redis 高可用部署、本地降级策略或活动暂停机制。

### Q64：如果你继续演进这个项目，下一步做什么？

我会优先做三件事：

1. 更系统的压测和性能分析
2. 缓存治理继续增强，比如逻辑过期或 Bloom Filter
3. 消息重试、死信和补偿链路继续完善

### Q65：如果面试官让你总结这个项目最体现你后端能力的地方，你怎么说？

我会说，这个项目最体现我的地方不是“会用很多中间件”，而是能围绕一个真实问题逐步演进系统：从同步版秒杀开始，先解决库存和一人一单，再前置 Redis Lua，之后引入 RocketMQ 异步下单，最后补齐限流、幂等、缓存治理、可观测和压测。这个过程说明我不只是会写功能，还能系统性地思考高并发后端设计。

## 15. 流程题高频补充

这一部分专门补“面试官最喜欢按流程追问”的题型。  
这类问题通常不是问一个点，而是让你把一次请求从前到后讲完整。

### Q66：你介绍一下用户注册整个过程发生了什么？

可以按这条线讲：

1. 用户在前端输入用户名和密码，发起注册请求
2. API 先做基础参数校验，比如用户名长度、密码长度、是否为空
3. 服务层先查数据库里是否已存在同名用户
4. 如果用户已存在，直接返回 `USER_ALREADY_EXISTS`
5. 如果用户不存在，对密码做 `bcrypt` 哈希，而不是明文存储
6. 把用户名和密码哈希写入 `users` 表
7. 注册成功后我会直接签发 JWT，减少用户再走一次登录流程
8. 返回 token、过期时间和用户基础信息

如果面试官继续追问，我会强调：  
注册的关键点不是“写一条用户数据”，而是参数校验、重复用户名判断、密码安全存储以及注册成功后的登录态建立。

### Q67：你介绍一下用户登录整个过程发生了什么？

可以这样讲：

1. 用户输入用户名和密码，调用登录接口
2. API 做参数校验
3. 服务层根据用户名查 `users` 表
4. 查到用户后，用 `bcrypt.CompareHashAndPassword` 对用户输入密码和数据库中的哈希做比对
5. 如果密码不对，返回 `INVALID_CREDENTIALS`
6. 如果密码正确，生成 JWT
7. 返回 token、过期时间和用户信息
8. 后续用户访问受保护接口时，把 token 放在 `Authorization: Bearer ...` 里
9. 中间件解析 token，提取 `user_id` 和 `username` 放到上下文里，业务接口再继续执行

### Q68：用户登录成功后，再访问一个受保护接口时中间发生了什么？

以 `GET /api/v1/me` 为例：

1. 用户带着 JWT 请求接口
2. Gin 中间件先取 `Authorization` 头
3. 解析出 Bearer Token
4. 用 JWT 管理器校验签名、过期时间、issuer
5. 解析成功后，把用户信息写进 Gin context
6. 业务 handler 再从 context 里取当前用户
7. 查询用户数据并返回

这条流程的重点是：  
登录接口负责“签发 token”，鉴权中间件负责“解释 token”，真正的业务接口只消费当前用户上下文。

### Q69：为什么密码不能明文存储？你这里怎么处理的？

因为明文密码一旦数据库泄露，用户密码会直接暴露，风险很大。  
我这里用的是 `bcrypt`，它适合做密码哈希，具备加盐和抗暴力破解能力。数据库里只保存哈希值，登录时再做比对。

### Q70：如果用户点击登录后立刻关闭网页，会不会登录成功？

这个问题可以分情况回答。参考常见登录流程面试问法，我会这样说：

- 如果只是关闭当前网页，但浏览器进程还在，服务端处理完成后，浏览器仍有机会接收响应并保存登录凭证，理论上可能登录成功。
- 如果是直接关闭整个浏览器，浏览器网络进程也可能一起结束，服务端响应返回时客户端未必还能接收到，所以结果不一定成功。

在我这个项目里，JWT 是由接口返回给客户端的，如果客户端在响应返回前就彻底关闭，前端就拿不到 token，自然也无法建立登录态。

### Q71：如果有人用脚本批量注册账号怎么办？

当前项目的注册链路主要做了基础校验和唯一用户名约束，但如果要扩展到真实业务，我会补这几层：

1. 图形验证码 / 滑块验证码
2. IP 维度频率限制
3. 设备指纹或 User-Agent 异常检测
4. 同一 IP / 同一网段的短时间注册阈值控制
5. 昵称或用户名的敏感词检测

也就是说，当前项目核心重心在秒杀链路，不在注册风控，但这部分我能讲清楚怎么补。

### Q72：你介绍一下用户点“抢购”之后整个过程发生了什么？

这是项目里最核心的一道流程题，我建议这样讲：

1. 用户先登录并拿到 JWT
2. 用户点击抢购，请求到达 Gin API
3. JWT 中间件识别用户身份
4. 秒杀守卫中间件做限流和短时重复提交保护
5. 进入 `Redis + Lua` 秒杀准入脚本
6. Lua 原子完成活动状态判断、时间窗口判断、一人一单判断和库存预扣减
7. 如果 Lua 返回失败，API 直接返回，比如 `SOLD_OUT`、`RATE_LIMITED`、`DUPLICATE_ORDER`
8. 如果 Lua 返回成功，API 生成 `order_no`，把订单消息发到 RocketMQ
9. API 立刻返回 `queued`
10. consumer 异步消费消息，先做消息幂等，再扣数据库库存、创建订单
11. 订单落库成功后，更新订单状态缓存并失效活动视图缓存
12. 用户查询订单时，如果 consumer 已处理完成，返回 `200`；如果还在队列中，返回 `202 ORDER_PROCESSING`

### Q73：为什么你这里要先走 Redis Lua，再进 RocketMQ？

因为这两个组件解决的问题不同：

- Redis Lua 解决的是“谁能通过准入”
- RocketMQ 解决的是“通过准入后的请求怎么削峰填谷”

如果没有 Redis Lua，很多无效请求也会进入 MQ；如果没有 MQ，通过准入后的请求会同步打数据库，写压力还是会大。

### Q74：为什么抢购成功后不直接创建订单？

因为在高并发场景下，同步建单会让数据库成为瓶颈。  
秒杀接口最重要的是先快速完成准入判断并返回结果，而真正的订单落库可以交给异步 consumer 来平滑处理。

### Q75：用户抢购成功后马上查订单，为什么有时候查不到？

因为这是异步下单。API 只负责把请求写入 RocketMQ，consumer 再异步落库。在这段时间窗口里，MySQL 里还没有订单记录。

为了避免用户误以为失败，我加了订单状态缓存：

- 如果还没落库，但缓存里状态是 `queued`
- 那订单详情接口会返回 `202 ORDER_PROCESSING`

### Q76：你介绍一下用户查订单整个过程？

可以按这条线讲：

1. 用户带着 JWT 调用 `GET /api/v1/orders/{orderNo}`
2. 中间件识别当前用户
3. 服务层先查 MySQL 是否已有订单
4. 如果已经落库，直接返回订单详情
5. 如果数据库里没有，再查 Redis 的订单状态缓存
6. 如果缓存里是 `queued`，返回 `202 ORDER_PROCESSING`
7. 如果缓存里也没有，才返回 `404 ORDER_NOT_FOUND`

这条设计主要是为了适配异步下单场景下的用户体验。

### Q77：你介绍一下活动预热整个过程？

可以这样回答：

1. 活动创建完成后，管理员调用预热接口
2. 服务层从数据库查活动详情和库存信息
3. 把活动详情写入 `seckill:activity:{id}`
4. 把活动库存写入 `seckill:stock:{id}`
5. 把活动列表写入 `seckill:activity:list`
6. 后续用户访问活动列表或详情时，优先命中 Redis

预热的意义是把活动开始前的热点数据提前打入缓存，避免活动开始的第一波请求全去打数据库。

### Q78：如果 Redis 里没有预热数据怎么办？

我这里做了兜底：

- 秒杀 Lua 如果发现库存 key 不存在，会返回 `NOT_PREHEATED`
- 服务层会尝试把活动详情重新写入缓存，再重试一次 Lua

也就是说，预热是主路径，但不是唯一兜底路径。

### Q79：如果 RocketMQ 发消息失败怎么办？

如果发消息失败，我会做 Redis 补偿：

1. 把预扣减的库存加回去
2. 删除用户标记

这样可以避免“库存已经扣掉，但下单消息没发出去”的不一致。

### Q80：如果 consumer 消费失败怎么办？

当前 RocketMQ 会触发重新投递，consumer 下一次还能继续消费。  
而且在 consumer 端我先做消息幂等，再进入扣库存和建单事务，所以重复消费不会造成重复扣库存。

### Q81：如果数据库扣库存成功，但订单写入失败怎么办？

这类场景在我当前实现里由同一个数据库事务兜底。  
也就是说，库存更新和订单插入是在同一事务里完成的，只要订单插入失败，事务就会回滚，库存不会被单独扣掉。

### Q82：如果 Redis 挂了，你这个秒杀系统会怎样？

当前架构下，Redis 是秒杀入口强依赖组件。如果 Redis 不可用，秒杀准入会受影响，系统不能继续正常做高并发秒杀。

我会这样回答：

- 这是高性能设计下的高依赖权衡
- 真正上线场景里应当用 Redis 高可用方案
- 同时加监控、降级开关和活动暂停机制

### Q83：如果 RocketMQ 挂了呢？

API 在生产消息失败时会做 Redis 补偿，所以不会出现“准入成功但消息丢失”的问题。  
但异步下单能力会暂时不可用，这时系统要么快速失败，要么降级成同步链路，这取决于业务要求。

### Q84：如果 MySQL 压力过大怎么办？

我的项目里已经做了几层缓冲：

1. 热点读流量先打 Redis
2. 秒杀准入在 Redis Lua 里完成
3. 通过准入的请求再进 RocketMQ 异步落库

如果数据库还是有压力，下一步可以考虑：

- 分库分表
- 读写分离
- 更严格的限流
- 把部分查询进一步缓存化

### Q85：为什么你项目里要专门做缓存穿透 / 击穿 / 雪崩？

因为面试官不会只满足于“你用了 Redis”，他们更关心你有没有考虑缓存在高并发下的边界问题。

我现在项目里已经落了：

- 空值缓存防穿透
- TTL 随机抖动防雪崩
- `singleflight` 防热点 key 击穿

所以我可以明确说，我不是只会“把数据放进 Redis”，而是考虑了热点缓存真正会遇到的问题。

### Q86：如果面试官让我从用户视角讲完整业务链路，我怎么收尾？

我会这样总结：

用户先登录，登录成功后拿到 JWT；浏览活动页时，大部分热点数据来自 Redis；点击抢购后，系统先在 Redis Lua 中做原子准入校验，通过后把订单消息发到 RocketMQ 并立刻返回 `queued`；consumer 异步落库后，用户再查订单就能看到最终结果；如果查得太早，也能拿到 `ORDER_PROCESSING` 而不是直接失败。

这个总结的好处是：  
你不是在背中间件，而是在讲一个完整的、用户真实能感知到的后端流程。
