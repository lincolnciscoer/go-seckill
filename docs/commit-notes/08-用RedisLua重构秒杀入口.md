# 第 08 次提交：用 Redis Lua 重构秒杀入口

## 本次目标

把秒杀入口从“直接打数据库”升级成“先过 Redis Lua 原子校验，再进入后续下单逻辑”。

这一提交要完成：

1. 用 Lua 原子校验活动状态、时间窗口、重复下单和库存
2. 在 Redis 中先做库存预扣减
3. 数据库下单失败时，对 Redis 做补偿
4. 秒杀成功后只失效活动视图缓存，保留库存和用户标记

## 我这次做了什么

1. 增加了 Lua 脚本秒杀闸门
2. 增加了 Redis 补偿脚本
3. 秒杀服务改成：
   - 先加载活动信息
   - 再执行 Lua 原子校验与预扣减
   - 最后执行数据库下单
4. 如果数据库下单失败，会回滚 Redis 中的库存和用户标记
5. 活动创建后会主动失效相关缓存，避免列表脏读

## 为什么这一步很重要

这是秒杀系统第一次真正具备“高并发入口前置闸门”的形态。

虽然现在后段还是同步下单，但前段已经开始像真正的秒杀系统那样工作：

- 先在 Redis 中挡住无效请求
- 再把少量通过准入的请求送到数据库

这会显著减少数据库在高并发入口处承受的压力。

### 2. 手动验证

1. 创建一个已开始的秒杀活动，库存设为 `1`
2. 用户 A 第一次秒杀，应成功
3. 检查 Redis：
   - 库存 key 应存在
   - 用户标记 key 应存在
4. 用户 A 再次秒杀，应被拦截
5. 用户 B 秒杀，应被售罄拦截
6. 再查活动详情，确认库存为 `0`

## 本次实际验证结果

- `go test ./...`：通过
- 预热后，Redis 中库存 key 初始值为 `1`
- 首次秒杀成功后，Redis 中库存 key 变为 `0`
- 首次秒杀成功后，用户标记 key 成功写入 Redis
- 同一用户再次秒杀：返回 HTTP 409
- 第二个用户在库存耗尽后秒杀：返回 HTTP 409
- `GET /api/v1/activities/{id}`：返回 HTTP 200，库存为 `0`
