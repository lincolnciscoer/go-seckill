## 本次目标

把异步秒杀链路从“能跑”继续推进到“更像真实系统”。

这一提交要完成：

1. 用户 / IP / 活动维度限流
2. 秒杀接口的重复提交保护
3. 订单处理中状态缓存
4. 异步下单时，订单详情接口返回更合理的语义

## 我这次做了什么

1. 增加了 Redis 版速率限制器
2. 增加了秒杀守卫中间件
   - 用户维度限流
   - IP 维度限流
   - 活动维度限流
   - 短时重复提交保护锁
3. 增加了订单状态缓存
   - 秒杀请求入队时写入 `queued`
   - consumer 落库成功后写入 `created`
4. 调整订单详情接口语义
   - 如果订单还没落库，但状态缓存里是 `queued`
   - 接口返回 HTTP `202`
   - 错误码为 `ORDER_PROCESSING`
5. 补充了新的错误码
   - `RATE_LIMITED`
   - `DUPLICATE_SUBMIT`
   - `ORDER_PROCESSING`

## 为什么订单处理中状态很重要

异步下单的一个典型问题是：

- 用户已经抢购成功
- 但 consumer 还没来得及落库
- 这时如果直接查订单详情，接口只会返回 `404`

这会让用户误以为下单失败。

所以更合理的做法是：

- 用 Redis 记录订单当前状态
- 在订单真正落库前，先告诉用户“处理中”
