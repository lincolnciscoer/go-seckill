# 第 07 次提交：实现订单领域与同步版秒杀闭环

## 本次目标

让系统第一次具备“真正能下单”的能力。

这一提交要完成：

1. 订单模型和订单查询接口
2. 用原生 SQL 实现库存扣减与订单创建
3. 同步版秒杀接口
4. 重复下单、库存不足、活动时间窗口等规则校验

## 我这次做了什么

1. 增加了订单模型
2. 增加了基于 `database/sql` 的订单仓储
   - 事务扣减库存
   - 插入订单
   - 查询订单详情
   - 查询用户订单列表
3. 增加了 `OrderService`
4. 增加了 `SeckillService`
   - 校验活动状态
   - 校验活动时间窗口
   - 校验重复下单
   - 执行同步下单
5. 增加了接口：
   - `POST /api/v1/seckill/activities/:id/attempt`
   - `GET /api/v1/orders/:orderNo`
   - `GET /api/v1/orders/me`
6. 秒杀成功后会主动失效活动缓存，避免库存缓存脏读

## 为什么这里开始用原生 SQL

因为库存扣减和订单创建已经进入热点写路径了。

这类路径最重要的是：

- 事务边界清晰
- SQL 行为可控
- 锁粒度和更新条件容易看懂

所以这一层我没有继续沿用 GORM，而是切到了原生 SQL。

### 1. 自动化测试


### 2. 手动验证

1. 注册并登录用户 A
2. 创建商品和一个已开始的秒杀活动，库存设为 `1`
3. 用户 A 发起秒杀，应成功创建订单
4. 用户 A 再次发起秒杀，应返回重复下单
5. 注册并登录用户 B
6. 用户 B 再次发起秒杀，应返回售罄
7. 查询订单详情和我的订单列表
8. 查询活动详情，确认剩余库存已变为 `0`

## 本次实际验证结果

- `go test ./...`：通过
- `POST /api/v1/seckill/activities/{id}/attempt`：首次请求返回 HTTP 200，成功创建订单
- 同一用户再次秒杀：返回 HTTP 409
- 第二个用户在库存耗尽后秒杀：返回 HTTP 409
- `GET /api/v1/orders/{orderNo}`：返回 HTTP 200，可查询订单详情
- `GET /api/v1/orders/me`：返回 HTTP 200，可查询当前用户订单列表
- `GET /api/v1/activities/{id}`：返回 HTTP 200，秒杀后剩余库存已更新为 `0`
