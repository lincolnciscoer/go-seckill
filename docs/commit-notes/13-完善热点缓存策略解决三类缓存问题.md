# 第 13 次提交：完善热点缓存策略，解决三类缓存问题

## 本次目标

把活动缓存链路从“有缓存”升级成“有热点缓存治理能力”。

这一提交要完成：

1. 缓存穿透：空值缓存
2. 缓存雪崩：TTL 随机抖动
3. 缓存击穿：`singleflight` 防止热点 key 重建时并发打库

## 我这次做了什么

1. 在活动详情缓存中增加空值缓存
   - 对不存在的活动 ID，缓存一个短 TTL 的空标记
   - 避免不存在 ID 被反复访问时持续打 MySQL
2. 在活动详情、活动列表、库存缓存中加入 TTL 随机抖动
   - 默认 TTL 基础上增加随机时间
   - 避免大量 key 同时过期造成缓存雪崩
3. 在活动列表和活动详情的缓存重建路径中加入 `singleflight`
   - 同一个热点 key 在缓存 miss 时，只允许一次回源数据库
   - 其他并发请求复用同一次重建结果
4. 为这三项补充了自动化测试

## 这三类问题分别怎么解决

### 1. 缓存穿透

问题：

- 请求查询一个根本不存在的活动 ID
- 缓存一直 miss
- 每次都直接打数据库

解决：

- 对不存在的活动写入空值缓存
- 下次查询时直接命中空值，不再回源数据库

### 2. 缓存雪崩

问题：

- 很多缓存 key 使用相同 TTL
- 在同一时间集中失效
- 大量请求瞬间一起打数据库

解决：

- 在缓存 TTL 上增加随机抖动
- 让 key 的实际过期时间错开

### 3. 缓存击穿

问题：

- 某个特别热的 key 过期
- 大量并发请求一起回源数据库

解决：

- 使用 `singleflight` 合并同一个 key 的重建请求
- 让并发请求只触发一次数据库查询
